  <action>
  **1. Extend WizardContext in types.ts:**
  Add `slackConnected?: boolean;` field to WizardContext interface (after the `gitDirty` field). This tracks whether Slack was connected for the Phase 7 success summary.

  **2. Create packages/ts/liftoff/src/steps/setup-slack.ts:**

  Export `setupSlackStep` implementing the Step interface. Follow the exact pattern from setup-mcp.ts.

  **shouldRun guards (SLK-05):**
  - Return false if `ctx.keyProvided` (no user identity — same as MCP step)
  - Return false if `!ctx.accessToken` (auth step didn't run or failed)
  - Return false if `ctx.completedSteps.includes("setup-slack")` (idempotency)
  - Return true otherwise

  **run implementation:**

  a) **Prompt user (SLK-01):** Use `p.confirm()` from @clack/prompts:
     ```
     message: "Would you like to set up Slack alerts?"
     ```
     If cancelled or false, return `{ status: "skipped", message: "Slack setup skipped" }`.

  b) **Show explanation:** Before opening browser, use `p.log.info()` to explain:
     "Slack alerts notify your team when agent failures are detected.\nWe'll open your browser to connect a Slack workspace."

  c) **Slack OAuth flow (SLK-02):** This is the CLI-to-browser OAuth relay, NOT the web-based flow from the context repo. The CLI handles the entire flow locally:

     - Generate a random state string: `crypto.randomBytes(16).toString("hex")`
     - Start localhost callback server: `const { port, waitForCallback, close } = await startCallbackServer(state, 60_000)` (60s timeout — Slack OAuth takes longer than simple auth)
     - Build the Slack OAuth URL. The CLI opens the browser DIRECTLY to Slack's OAuth authorize URL (not to the context repo's /api/integrations/slack/authorize). Build the URL:
       ```
       https://slack.com/oauth/v2/authorize?client_id=${clientId}&scope=channels:history,channels:read,chat:write,chat:write.public,commands,channels:join,groups:read,im:history,im:read&redirect_uri=http://127.0.0.1:${port}/callback&state=${state}
       ```
       The `client_id` comes from the context repo's API. Create a helper that fetches it:
       - Call `GET ${baseUrl}/api/cli/slack-client-id` with Authorization: Bearer ${ctx.accessToken}
       - This endpoint returns `{ clientId: string }`.
       - The baseUrl should be `https://www.thecontext.company` (same as other CLI endpoints).
       - If this fetch fails, log a warning and return skipped.

     **IMPORTANT:** The CLI redirect_uri points to `http://127.0.0.1:${port}/callback` — the local server. Slack redirects the browser there after the user authorizes. The localhost-server.ts handles receiving the code+state.

     - Open browser with `open` package (already used in Phase 2 for auth): `const openModule = await import("open"); await openModule.default(slackOAuthUrl);`
     - Show spinner: `const s = p.spinner(); s.start("Waiting for Slack authorization...");`
     - Wait: `const result = await waitForCallback();`
     - Stop spinner: `s.stop();`

  d) **Handle callback result:**
     - If `result === null` (timeout or bad state): `close()`, log warning "Slack authorization timed out or was cancelled", return `{ status: "skipped", message: "Slack auth timed out" }`
     - If result has code: Exchange the code on the SERVER side. The CLI cannot have the Slack client secret. Send the code to the context repo:
       ```
       POST ${baseUrl}/api/cli/slack-callback
       Authorization: Bearer ${ctx.accessToken}
       Body: { code: result.code, redirectUri: `http://127.0.0.1:${port}/callback` }
       ```
       This endpoint does the token exchange with Slack and stores the integration (reusing existing context repo logic). Returns `{ ok: true, teamName: string }` on success or `{ ok: false, error: string }` on failure.

  e) **Handle exchange result:**
     - Success: `p.log.success(`Connected to Slack workspace: ${teamName}`)`, set `ctx.slackConnected = true`
     - Failure: `p.log.warn("Could not complete Slack setup: " + error)`, return `{ status: "skipped", message: "Slack token exchange failed" }`

  f) **Post-connect guidance (SLK-03):** After successful connection, show:
     ```
     p.log.info(
       pc.cyan("Next steps for Slack alerts:\n") +
       "1. Add the Context Company bot to a channel\n" +
       "2. Type /subscribe in that channel to start receiving alerts"
     );
     ```

  g) Push "setup-slack" to `ctx.completedSteps`, return `{ status: "completed" }`.

  **cleanup method:** Call `close()` on the callback server if it's still running. Store the close function in a module-level variable so cleanup can access it.

  **Note on SLK-04 (no Pro gate):** This was already handled in Phase 1 (SRV-03 removed the Pro plan requirement from the context repo). The CLI step simply connects — no plan check needed.

  **Server endpoints needed:** This step requires two NEW endpoints in the context repo:
  - `GET /api/cli/slack-client-id` — returns the Slack client ID (so CLI doesn't hardcode it)
  - `POST /api/cli/slack-callback` — receives the OAuth code, does server-side token exchange with Slack client secret, stores the integration

  These endpoints must be created in the context repo (same location as existing /api/cli/auth and /api/cli/keys). Create them at:
  - `/Users/rohil/Documents/Programming/tcc/context/demo/src/app/api/cli/slack-client-id/route.ts`
  - `/Users/rohil/Documents/Programming/tcc/context/demo/src/app/api/cli/slack-callback/route.ts`

  **GET /api/cli/slack-client-id/route.ts:**
  - Authenticate via WorkOS accessToken (same pattern as /api/cli/keys)
  - Return `NextResponse.json({ clientId: env.SLACK_CLIENT_ID })`

  **POST /api/cli/slack-callback/route.ts:**
  - Authenticate via WorkOS accessToken
  - Extract `code` and `redirectUri` from request body
  - Exchange code for token: POST to `https://slack.com/api/oauth.v2.access` with `{ code, client_id: env.SLACK_CLIENT_ID, client_secret: env.SLACK_CLIENT_SECRET, redirect_uri: redirectUri }`
  - Validate response: check `tokenData.ok`, extract teamId, teamName, botUserId, access_token
  - Encrypt and store using same pattern as existing callback route (encryptToken, postgresDb insert into slackIntegrations with orgId from authenticated user)
  - Return `{ ok: true, teamName }` on success, `{ ok: false, error: "..." }` on failure
